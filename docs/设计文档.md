````markdown
# Cloudflare 临时邮箱基础设施项目需求文档

## 1. 项目定位

我要开发一个基于 Cloudflare 的临时邮箱后端基础设施项目。

这不是普通的“用户打开页面后随机分配一个临时邮箱”的网站。

这个系统主要作为后端基础设施使用：其他 Worker、后端服务、业务系统可以通过 API 创建临时邮箱，系统返回邮箱地址和加密访问链接。终端用户拿到加密访问链接后，可以查看该邮箱收到的邮件列表、邮件详情和附件。

参考前端风格：https://mail.55store.vip/  
界面要求简洁、干净、轻量。

---

## 2. 核心技术栈

使用 Cloudflare 全家桶：

- Cloudflare Workers
- Cloudflare Email Worker / Email Routing
- Cloudflare D1
- Cloudflare R2
- Cloudflare KV
- Wrangler
- TypeScript
- Hono 或原生 Worker Router
- 前端可以使用 React / Vite / Worker SSR

## 3. 核心功能

### 3.1 临时邮箱创建

系统用户或外部服务通过 API 创建临时邮箱。

创建后返回：

- 邮箱地址
- 过期时间
- encrypted access link

示例：

```json
{
  "email_address": "a8x92k@tmp.example.com",
  "expires_at": "2026-05-09T12:00:00Z",
  "encrypted_access_url": "https://mail.example.com/inbox/xxxxx"
}
````

### 3.2 邮箱访问

终端用户不需要登录。

终端用户通过 encrypted access link 访问邮箱：

```text
/inbox/:encryptedToken
```

用户可以看到：

* 邮箱地址
* 邮箱过期时间
* 邮件列表
* 邮件详情
* 附件下载

邮箱页面要求：

* 每 30 秒自动刷新
* 支持手动刷新
* 邮件默认保存 24 小时
* 邮箱默认保存 24 小时
* 支持创建时传入自定义 ttl_seconds

## 5. Cloudflare 架构要求

### 5.1 Worker 运行时资源访问

Worker 访问 D1 / R2 / KV 必须使用 Cloudflare bindings。

Worker 运行时访问 D1 / R2 / KV 不允许使用 Cloudflare 用户 token，也不允许依赖 Cloudflare API Token。

示例：


* 用户

* 邮箱元数据
* 邮件元数据
* 加密访问链接元数据
* 审计日志
* 系统设置
* Cloudflare 集成状态

R2：

* 原始邮件 MIME
* text 正文
* html 正文
* 附件

KV：

* 短期缓存
* 邮箱最新状态
* 轮询优化
* rate limit
* access token 校验缓存

## 6. Cloudflare Token / Bindings / Secrets 设计

项目中必须区分以下几类东西。

### 6.1 D1 / R2 / KV Bindings

D1 / R2 / KV 通过 `wrangler.jsonc` 或 `wrangler.toml` 配置。

Worker 运行时直接通过 env 访问。

这部分不需要 Cloudflare API Token。

### 6.2 Workers Builds / 部署 API 令牌

创建 Worker / Workers Builds 时，可以选择一个 Cloudflare API 令牌。

这个令牌用于：

* 构建
* 上传 Worker
* 部署 Worker
* Wrangler 部署阶段调用 Cloudflare API
* 首次部署时 automatic provisioning D1 / KV / R2

这个 token 是部署用的，不是普通用户业务 token。

### 6.3 Runtime Cloudflare API Token

后台管理员需要自动配置子域名 / DNS / Email Routing 时，Worker 后端需要调用 Cloudflare API。

这个 token 应该在创建 Worker / 部署阶段作为 runtime secret 绑定，而不是在后台页面要求管理员填写。

推荐 secret 名称：

```text
CLOUDFLARE_API_TOKEN
```

Worker 后端通过：

```ts
env.CLOUDFLARE_API_TOKEN
```

调用 Cloudflare API。

后台页面不展示、不返回、不保存明文 token。

### 6.4 系统业务 API Token

普通用户可以创建自己的业务 API token，用于调用：

```text
POST /api/v1/mailboxes
```

业务 API token：

* 只显示一次
* D1 只存 hash
* 可以禁用
* 可以删除
* 可以记录 last_used_at
* 只能操作 token 所属用户的资源

## 7. Wrangler 配置优先原则

所有能放进 `wrangler.jsonc` 或 `wrangler.toml` 的配置，都应该放进 Wrangler 配置。

不要用 setup 脚本手动创建 D1 / KV / R2 作为主流程。

优先使用 Wrangler automatic provisioning。

### 7.1 Wrangler 配置应包含

* Worker name
* main entry
* compatibility_date
* D1 binding
* KV binding
* R2 binding
* 普通 vars
* secrets required 声明
* cron triggers
* routes / custom domain，如需要

### 7.2 D1 / KV / R2 配置

在 `wrangler.jsonc` 里只声明 binding。

尽量不要在初始模板里写死：

* database_id
* kv namespace id
* bucket_name

首次部署时让 Wrangler 自动创建资源。

示例 `wrangler.jsonc`：

```jsonc
{
  "$schema": "./node_modules/wrangler/config-schema.json",
  "name": "temp-mail",
  "main": "src/index.ts",
  "compatibility_date": "2026-05-08",

  "workers_dev": true,

  "d1_databases": [
    {
      "binding": "DB"
    }
  ],

  "kv_namespaces": [
    {
      "binding": "MAIL_KV"
    }
  ],

  "r2_buckets": [
    {
      "binding": "MAIL_R2"
    }
  ],

  "vars": {
    "APP_NAME": "Temp Mail",
    "DEFAULT_MAIL_TTL_SECONDS": "86400",
    "CLOUDFLARE_ACCOUNT_ID": "",
    "CLOUDFLARE_ZONE_ID": "",
    "CLOUDFLARE_ZONE_NAME": ""
  },

  "secrets": {
    "required": [
      "CLOUDFLARE_API_TOKEN",
      "SESSION_SECRET",
      "LINK_SECRET"
    ]
  },

  "triggers": {
    "crons": [
      "*/15 * * * *"
    ]
  }
}
```

## 8. 部署设计

### 8.1 不需要主 setup 脚本

不要求写 `setup-cloudflare.mjs` 作为主部署流程。

主部署流程应该依赖：

* `wrangler.jsonc` / `wrangler.toml`
* Workers Builds
* Wrangler automatic provisioning
* package.json scripts
* Workers Builds deploy command

### 8.2 D1 migrations

D1 migrations 不放 setup 脚本。

D1 migrations 写到 `package.json` 的 deploy script 或 Workers Builds 的 deploy command 里。

第一次部署时，D1 可能还没有被 automatic provisioning 创建，所以第一次部署可以先 deploy，再 migrations。

推荐 package scripts：

```json
{
  "scripts": {
    "dev": "wrangler dev",
    "deploy:first": "wrangler deploy && wrangler d1 migrations apply DB --remote",
    "deploy": "wrangler d1 migrations apply DB --remote && wrangler deploy",
    "db:migrate": "wrangler d1 migrations apply DB --remote",
    "deploy:worker": "wrangler deploy"
  }
}
```

首次部署使用：

```bash
npm run deploy:first
```

后续部署使用：

```bash
npm run deploy
```

### 8.3 Runtime secrets

Runtime secrets 不写入 git。

不要把 Cloudflare API Token 写进：

* `wrangler.jsonc`
* `wrangler.toml`
* `.env`
* D1
* 前端代码

使用 Cloudflare Dashboard 的 Variables & Secrets 或 Wrangler secret 设置：

```bash
npx wrangler secret put CLOUDFLARE_API_TOKEN
npx wrangler secret put SESSION_SECRET
npx wrangler secret put LINK_SECRET
```

### 8.4 可选辅助脚本

可以写一个可选脚本，只用于本地辅助：

* 检查 wrangler 是否安装
* 执行 wrangler whoami
* 显示当前 Cloudflare 账号
* 生成随机 SESSION_SECRET / LINK_SECRET
* 输出部署说明

但该脚本不是主部署流程，不负责创建 D1 / KV / R2。

## 9. Worker 组织方式

优先做成一个 Worker，同时处理 Web/API 和 Email。

一个 Worker 里同时实现：

```ts
export default {
  async fetch(request, env, ctx) {
    // Web 页面、后台、API、加密链接访问
  },

  async email(message, env, ctx) {
    // Email Worker 收邮件
  },

  async scheduled(event, env, ctx) {
    // 定时清理过期邮箱、邮件、R2 对象
  }
}
```

这样可以让 Web/API 和 Email Worker 共用同一套：

* D1 binding
* R2 binding
* KV binding
* secrets
* vars

避免拆成多个 Worker 后自动 provision 出多套 D1 / KV / R2。

后续需要扩展时，再考虑拆分成：

* Web/API Worker
* Email Worker

拆分后必须保证两个 Worker 共享同一套 D1 / KV / R2 资源 ID。

## 10. 邮件接收设计

使用 Cloudflare Email Routing + Email Worker。

管理员后台配置一个收信子域名，例如：

```text
tmp.example.com
```

Cloudflare Email Routing 将该子域名的 catch-all 或 routing rule 指向 Email Worker。

创建具体邮箱时，不要每次调用 Cloudflare API。

正确流程：

```text
管理员配置 tmp.example.com
→ Email Routing catch-all 指向 Email Worker
→ 用户创建 a123@tmp.example.com
→ 只写入 D1 mailboxes 表
→ 邮件发到 a123@tmp.example.com
→ Email Worker 收到邮件
→ 查 D1 判断邮箱是否存在、是否启用、是否过期
→ 有效则保存邮件
```

Email Worker 收到邮件后：

1. 读取 `message.to`
2. 查询 D1
3. 判断邮箱是否存在
4. 判断邮箱是否启用
5. 判断邮箱是否过期
6. 无效邮箱拒收或丢弃
7. 有效邮箱写入 D1 messages 元数据
8. 原始 MIME 存 R2
9. text/html 正文存 R2
10. 附件存 R2
11. KV 写入最新状态缓存

## 11. 后台 Cloudflare 子域名配置

管理员后台支持添加子域名。

后台添加子域名时：

1. 检查当前登录用户是否为 admin
2. 检查 runtime env 是否存在 `CLOUDFLARE_API_TOKEN`
3. 检查系统是否配置 `CLOUDFLARE_ACCOUNT_ID`
4. 检查系统是否配置 `CLOUDFLARE_ZONE_ID`
5. 校验输入的子域名是否属于当前 zone
6. 调 Cloudflare API 创建或验证 DNS
7. 调 Cloudflare API 配置 Email Routing subdomain / routing rule
8. 将 catch-all 或 rule 指向当前 Email Worker
9. 将域名写入 D1 domains 表
10. 记录 cloudflare_rule_id / dns_record_id
11. 写入 audit_logs
12. 返回配置状态

如果 Cloudflare API 失败，后台要显示明确错误，并给出人工配置指引。

普通用户不能调用任何 Cloudflare 配置接口。

部署 / 自动配置可能需要以下权限：

```text
Account Workers Scripts Edit
Account D1 Edit
Account R2 Edit
Account Workers KV Storage Edit
Zone Read
Zone DNS Edit

```

其中：

* D1 / R2 / KV 创建主要用于部署阶段
* DNS / Email Routing 主要用于后台自动配置子域名
* Workers Routes 用于自动绑定 route / custom domain

后台只能操作预设 zone，不允许普通请求传任意 zone_id 去操作其他域名。

## 13. Encrypted Access Link 设计

创建邮箱时生成：

* mailbox_id
* access_secret
* encryptedToken

返回链接：

```text
/inbox/:encryptedToken
```

encryptedToken 不能直接暴露真实 mailbox_id。

token 内容应包含：

* mailbox_id
* expires_at
* nonce
* signature 或加密数据

可以使用：

* HMAC 签名
* AES-GCM 加密
* Web Crypto API

服务端验证 token 后才允许访问邮箱。

要求：

* 链接默认和邮箱一样 24 小时过期
* 支持自定义 ttl_seconds
* 访问者不需要登录
* 链接只能访问对应 mailbox
* 不能横向访问其他邮箱
* token hash 可以存入 mailbox_access_links 表
* 可记录 last_used_at

## 15. D1 数据库设计

至少包含以下表。

### users

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  username TEXT,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('admin', 'user')),
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'disabled')),
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  last_login_at TEXT
);
```

### domains

```sql
CREATE TABLE domains (
  id TEXT PRIMARY KEY,
  domain TEXT NOT NULL UNIQUE,
  type TEXT NOT NULL CHECK (type IN ('root', 'subdomain')),
  zone_id TEXT,
  status TEXT NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'active', 'failed', 'disabled')),
  cloudflare_rule_id TEXT,
  cloudflare_dns_record_id TEXT,
  created_by TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (created_by) REFERENCES users(id)
);
```

### user_domains

```sql
CREATE TABLE user_domains (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  domain_id TEXT NOT NULL,
  created_at TEXT NOT NULL,
  UNIQUE(user_id, domain_id),
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (domain_id) REFERENCES domains(id)
);
```

### api_tokens

```sql
CREATE TABLE api_tokens (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  name TEXT NOT NULL,
  token_hash TEXT NOT NULL UNIQUE,
  token_prefix TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'revoked')),
  last_used_at TEXT,
  created_at TEXT NOT NULL,
  revoked_at TEXT,
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### mailboxes

```sql
CREATE TABLE mailboxes (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  domain_id TEXT NOT NULL,
  email_address TEXT NOT NULL UNIQUE,
  local_part TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'disabled', 'expired')),
  access_secret_hash TEXT NOT NULL,
  created_by_token_id TEXT,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL,
  disabled_at TEXT,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (domain_id) REFERENCES domains(id),
  FOREIGN KEY (created_by_token_id) REFERENCES api_tokens(id)
);
```

### mailbox_access_links

```sql
CREATE TABLE mailbox_access_links (
  id TEXT PRIMARY KEY,
```


```text
rate_limit:inbox:{ip}:{window}
cloudflare:verify:{domain_id}
access_token_cache:{token_hash}
session:{session_id}
```

KV 只做缓存，不作为唯一可信数据源。

最终可信数据以 D1 为准。

## 18. 过期和清理策略

邮箱和邮件默认保存 24 小时。

所有相关数据都要有 expires_at。

通过 Worker Cron 定时清理：

* 过期 mailboxes
* 过期 messages
* 过期 mailbox_access_links
* 对应 R2 raw MIME
* 对应 R2 text/html
* 对应 R2 attachments
* 相关 KV cache

Cron 建议：

```text
*/15 * * * *
```

即每 15 分钟清理一次。

## 19. 安全要求

1. 密码必须 hash 存储，不能明文。
2. API token 只显示一次，D1 只存 hash。
3. session 使用 httpOnly secure cookie。
4. 后台接口必须鉴权。
5. admin 接口必须检查 role=admin。
6. 普通用户资源查询必须带 user_id 限制。
7. encrypted link 只能访问对应 mailbox。
8. 附件下载必须校验权限。
9. Cloudflare API Token 不返回给前端。
10. Cloudflare API Token 不写入 D1 明文。
11. Cloudflare API Token 不写入 wrangler.jsonc。
12. Cloudflare API Token 不写入 git。
13. 管理员 Cloudflare 配置操作必须写 audit_logs。
14. 邮件 HTML 展示必须防 XSS。
15. HTML 邮件建议使用 sanitize 或 sandbox iframe。
16. API 要有基础 rate limit。
17. 登录接口要有 rate limit。
18. 加密链接访问也要有 rate limit。
19. 普通用户不能越权访问其他用户邮箱。
20. 普通用户不能触发 Cloudflare API。

## 25. 关键要求总结

必须遵守：

1. 这不是随机临时邮箱网站。
2. 邮箱必须由登录用户或 API 创建。
3. 创建邮箱返回 encrypted access link。
4. 终端用户通过 encrypted access link 查看邮件。
5. D1 / R2 / KV 通过 Worker bindings 访问。
6. D1 / R2 / KV 不通过 Cloudflare API Token 访问。
7. 能放进 wrangler.jsonc / wrangler.toml 的配置都放进去。
8. D1 / KV / R2 优先使用 Wrangler automatic provisioning。
9. D1 migrations 放到 package.json deploy script 或 Workers Builds deploy command。
10. Cloudflare API Token 不在后台页面填写。
11. 创建 Worker / 部署阶段选择或绑定 Cloudflare API 令牌。
12. 后台自动配置子域名时，Worker 通过 runtime secret `CLOUDFLARE_API_TOKEN` 调 Cloudflare API。
13. 普通用户不能触发 Cloudflare API。
14. 管理员 Cloudflare 配置操作必须写 audit log。
15. 创建具体邮箱时只写 D1，不要每次调用 Cloudflare API。
16. Email Routing 子域名配置一次，具体邮箱由 Email Worker 查 D1 判断。
17. 邮件正文和附件存 R2，不要塞进 D1。
18. KV 只做缓存，不做唯一可信数据源。
19. 默认 TTL 是 24 小时。
20. 邮箱页面每 30 秒自动刷新，支持手动刷新。
